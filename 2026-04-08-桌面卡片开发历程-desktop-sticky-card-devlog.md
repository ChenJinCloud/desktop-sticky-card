# 一个晚上，从痛点到开源：Desktop Sticky Card 开发全记录

## 过去发生了什么

我的日常工作状态是多线程的。产品开发、内容创作、个人项目、生活事务同时在跑，互相穿插。在这种工作方式下，我一直有一个非常明确的习惯需求：我需要随时能看到所有待办事项，反复确认它们，才能判断自己当前的节奏对不对、位置在哪里。这不是简单的"怕忘事"，而是一种保持掌控感和安全感的方式——我没办法把事情规划完就放心地后台挂起，只盯眼前一件事。

在过去相当长的时间里，我尝试过多种方案来满足这个需求。我用过滴答清单的桌面便签功能，用过 Windows 11 自带的便笺应用，也试过一直开着一个 IDE 或者终端窗口来编辑 Markdown 格式的日志文件。这些工具各有各的能力，但都没有完全解决我的核心问题。

## 导致了什么问题

所有这些工具有一个共同的致命缺陷：它们都需要我主动去打开、去切换窗口才能看到内容。而一旦某个东西需要"主动想起来才去看"，它作为锚点的意义就大打折扣了。我需要的是余光就能扫到的东西，像学生时代贴在书桌角落的那张便签纸一样，不用刻意去找，它就在那里。

具体来说，滴答清单虽然功能丰富但操作层级较深，改个内容需要点好几步；Windows 便笺过于简陋，不支持任务勾选和结构化内容；而开着 IDE 编辑 Markdown 的方式窗口太大，写代码或开会的时候很容易就被其他窗口完全盖住了。这些方案不是不好用，而是它们在"始终可见"这一点上都不够彻底。

## 用过什么方法解决

2026 年 4 月 7 日晚上大约七点半，我决定自己动手做一个。整个开发过程完全通过与 Claude Code 的对话完成，从零开始，没有任何模板或脚手架。

第一版在大约十分钟内就跑起来了。我告诉 Claude 我需要一个可以固定在桌面上的小卡片，可以置顶悬浮在任何页面之上，内容通过编辑一个文件来更新。Claude 选择了 Python 的 tkinter 作为技术方案——零依赖，Windows 自带 Python 环境就能跑。第一版使用了 Catppuccin Mocha 暗色主题，实现了 Markdown 简易渲染，支持标题、任务列表、普通列表、水平线和斜体，通过每 500 毫秒轮询文件变化来自动刷新卡片内容，并且做了无边框可拖拽的窗口和置顶切换。

但第一版马上遇到了一个 Windows 平台的经典问题：使用 `overrideredirect(True)` 创建的无边框窗口在 Windows 上无法正常获取焦点，导致鼠标点击没有反应。Claude 先尝试了改用 `toolwindow` 属性的方案，结果产生了系统标题栏和自定义顶栏的双层结构，视觉上非常别扭。最终通过 `withdraw → deiconify → lift → focus_force` 的组合技巧解决了这个焦点问题，同时保留了干净的无边框外观。

紧接着我提出了视觉风格的要求——我希望它看起来像 Medium 的阅读体验。于是整个配色方案从暗色主题重新设计为白底 Georgia 衬线字体的明亮风格，使用 Medium 的品牌绿作为强调色，淡灰色顶栏搭配细边框，整体观感干净素雅。

到这一步，卡片的内容更新还完全依赖于我在 Claude Code 对话中让 AI 编辑文件。我提了一个关键问题："这个是否必然需要 AI？还是可以做一个我只是在终端对话然后更新的方式？"Claude 的回答很直接——完全不需要 AI。于是它帮我做了一个纯规则驱动的对话终端 `chat.py`：双击打开一个终端窗口，直接打字就是添加任务，输入"完成 1"就勾选第一个任务，输入"帮助"查看所有命令。不调用任何 API，没有任何 AI 成本。同时还做了一个 `card.py` 作为单命令 CLI 工具，以及 `start.bat` 一键启动脚本。

在实际使用中，问题接连暴露。start.bat 因为项目文件夹名称包含中文，在 cmd.exe 中遭遇了 GBK 与 UTF-8 的编码冲突，导致路径解析失败，用户输入的中文被 cmd 当作命令执行。排查后发现根因是 bat 文件本身包含中文注释，解决方案是移除所有 bat 文件中的中文字符，并且在 `chcp 65001` 改变代码页之前先用 `set` 命令保存路径。

随后我发现输入"1.完成桌面小卡片的设计开发；2.完成洗头洗澡敷面膜；3.完成量化 todos 的 mvp"这样的内容时，系统会把整段话作为一个任务录入，而实际上这是五个独立的任务。于是增加了多任务拆分逻辑，能够识别数字编号加分号分隔的模式并自动拆分。同时修复了新任务总是追加到文件末尾而不是插入到任务列表区域的位置问题。

功能层面的迭代也在快速推进。我发现添加任务时没有记录创建时间，于是加上了时间戳功能，格式为 `04/07 21:05` 这样精简的形式。接着我希望勾选完成时也能记录完成时间，并且可以将已完成的任务折叠起来只看未完成的——于是实现了 All/Todo 视图切换和完成时间戳。为了让时间戳的显示不那么死板，又加了一个 T 按钮来控制时间戳的显示和隐藏。

卡片的固定宽度在内容较长时会导致文本换行过多，于是实现了窗口可拖拽调整大小的功能，通过检测鼠标在窗口边缘的位置来判断是哪个方向的 resize。同时实现了卡片上的直接编辑功能：点击 Edit 按钮进入编辑模式，显示原始 Markdown 文本，支持 Ctrl+S 保存和 Esc 取消，编辑期间暂停文件轮询避免冲突。

视觉体验方面，我注意到卡片上的文字相比同屏幕的网页文字有一种模糊的颗粒感。这是 tkinter 在高 DPI 屏幕上的经典问题——默认不声明 DPI 感知，Windows 会对窗口做模糊缩放。解决方案是在程序启动最开头调用 `ctypes.windll.shcore.SetProcessDpiAwareness(2)` 声明 Per-Monitor DPI Aware。

一个白色卡片放在白色网页背景上很不显眼的问题促使我实现了主题系统。最终做了八套配色方案：明亮（Light）、暗黑（Dark）、三款马卡龙色系（Rose、Mint、Lavender）和三款莫兰蒂色系（Green、Blue、Rose）。整个 UI 的颜色管理统一通过 `self.t()` 方法获取当前主题的颜色值，切换主题时销毁整个 UI 重新构建。在此基础上又加了五档字号切换（XS/S/M/L/XL），所有文字元素——标题、正文、时间戳、编辑器——都跟随字号联动。

窗口的位置、大小、主题、字号、时间戳显示、折叠状态等用户偏好全部保存到 `.card-state.json` 文件中，关闭后再打开能恢复到上次的状态。为了防止进程被强杀时丢失设置，还加了每五秒自动保存的兜底机制。

在所有功能基本稳定之后，我让 Claude 对整个项目做了一次代码质量审查。审查发现最严重的问题是 `chat.py` 和 `card.py` 之间存在大约 140 行完全重复的代码——文件读写、任务查询、增删改、多任务拆分的逻辑在两个文件中各写了一遍。于是提取了 `cardlib.py` 作为共享模块，`chat.py` 的命令分发也从 12 个 if-elif 分支重构为声明式的 dispatch table，`card.py` 精简到了约 80 行。

最后一个 bug 出现在实际使用流程中：如果先手动打开了卡片，再运行 start.bat（它也会启动卡片），就会出现两个卡片窗口。通过 msvcrt 文件锁实现了单实例保护，第二次启动时检测到锁文件已被占用就自动退出。

## 结果如何

从晚上七点四十分的第一行代码，到凌晨零点十五分推送到 GitHub 完成开源，整个过程大约四个半小时。项目经历了从 v0.1.0 到 v0.10.1 共十二个版本迭代，每个版本都对应着一个具体的使用痛点或功能需求。最终的产品是一个零依赖、双击即用的桌面置顶待办卡片工具，代码总量约 1400 行，包含完整的 GUI 程序、共享逻辑库、对话终端、CLI 工具、三个启动脚本、README、CHANGELOG 和 banner 设计。

## 方案是什么

Desktop Sticky Card 是一个 Markdown 文件驱动的桌面置顶卡片。它的核心设计原则是：**内容始终可见，不需要任何主动操作就能看到待办全局**。技术选型上使用 Python 标准库 tkinter，不依赖任何第三方包，Windows 环境下双击 bat 文件即可运行。内容存储为纯 Markdown 文件，意味着任何文本编辑器、终端命令、甚至 AI 编程助手都可以直接修改卡片内容。

## 优势是什么

相比现有的待办工具，Desktop Sticky Card 的核心差异在于"始终可见"的彻底性。它不是一个需要打开的 App，而是一个钉在桌面上、悬浮在所有窗口之上的固定元素。八套主题配色确保在任何桌面背景下都能保持辨识度，五档字号适配不同屏幕尺寸和视力需求。文件驱动的架构意味着数据完全本地化，没有云同步、没有账号体系、没有隐私顾虑。零依赖的技术选型让它在任何装有 Python 的 Windows 机器上都能直接运行，不需要安装任何东西。

## 推进计划是什么

当前版本（v0.10.1）的功能已经满足日常使用需求。后续计划包括：录制高清的演示视频替换当前画质不足的 GIF；探索跨平台支持的可能性（macOS、Linux）；以及根据实际使用中发现的新痛点持续迭代。项目已在 GitHub 开源（github.com/ChenJinCloud/desktop-sticky-card），采用 MIT 协议。

## 成本如何

整个项目的直接成本仅为一个晚上的时间和 Claude Code 的 API 用量。不依赖任何付费服务或商业组件，运行时零成本。对于使用者而言，唯一的前置条件是 Windows 10/11 系统和 Python 3.8 以上环境（Windows 默认包含 tkinter）。
